技术架构报告

Web/飞书远程继续与新建 Claude Code 对话

本报告说明 llm-proxy 2.0.0 远程对话能力的背景、技术调研、架构设计和核心实现。 结论是：当前发布版以 claude -p stream-json 作为稳定主链路， 同时保留 MCP channel 作为 Claude Code channels 可用后的增强链路。

版本 2.0.0
报告日期 2026-07-07
当前主路径 deliveryMode = cli
目标平台 Web Dashboard / 飞书
本机 Claude Code 飞书自建应用长连接 单卡片进度 + 最终文本 MCP channel 预留

0. 摘要结论

一句话方案

Web/飞书把远程消息送到 llm-proxy，服务端按 thread/cwd 选择或创建远程对话， 当前默认通过 claude -p --output-format stream-json 在本机项目目录执行， 将 stream-json 和 hook 事件聚合成进度快照，Dashboard 用 SSE 展示，飞书用一张交互卡片持续更新， 最终答案再以普通聊天文本推回。

为什么不是 channel 主链路

MCP channel 是架构上更接近“把外部消息注入正在运行的 Claude Code session”的方案， 但 Claude Code channels 仍处在 research preview，且本机实际启动时出现过 Channels are not currently available。发布版需要可用性优先，所以用 CLI fallback 作为主路径，channel 作为 auto 或 channel 模式保留。

当前发布策略

v2.0.0 的默认策略是“先稳定可用，再平滑升级”：飞书输入、卡片进度、最终文本、Web Dashboard 都基于 CLI fallback 完成；当 Claude Code channels 在用户环境里可用时，切换到 deliveryMode = auto，在线 channel instance 会优先接管，没有在线 instance 时继续回退 CLI。

1. 背景

这个项目原本已经具备 Claude Code 代理、hook 采集和 Dashboard 展示能力。远程对话的需求来自一个很具体的使用场景： 用户不一定坐在本机终端前，但希望通过 Web 或飞书把任务投递给本机 Claude Code，让它在真实项目目录、真实本机环境和现有代理配置下执行。

业务场景

  • 飞书私聊里发送普通消息，继续最近活跃远程 thread。
  • 飞书群里 @bot 或回复 bot 消息，创建或继续指定任务。
  • Dashboard 项目卡或 session list 里直接新建或继续远程对话。
  • 执行过程中能看到状态、工具摘要、错误和最终结果。

现有基础

  • Proxy server 已经拥有本地 HTTP 服务、配置管理和 SQLite。
  • Hooks 已经能捕获 Claude Code session、tool、notification、stop 事件。
  • Dashboard 已经有 SSE 和会话视图，可承载远程执行状态。
  • 通知系统已有飞书 webhook 输出，但不能接收用户消息。

关键差异

  • 远程对话是双向链路，不只是 hook 通知。
  • 需要 thread、cwd、session、消息投递状态的持久映射。
  • 飞书必须使用自建应用事件接收，不能依赖自定义 webhook 机器人。
  • 不能展示 Claude 隐藏思考，只展示可观察过程和结果。

目标与约束

功能目标
  • 支持 Web/飞书新建 Claude Code 远程对话。
  • 支持按 thread 继续远程对话，并尽量复用 Claude session。
  • 飞书显示单卡片进度，最终答案像聊天一样回到主会话。
  • 支持权限请求 relay 的架构预留。
  • 保留 MCP channel 路径，等待 Claude Code channels 可用。
非功能约束
  • 远程 API 必须有 token 或同源边界保护。
  • cwd 必须被 allowlist 限制，避免任意目录执行。
  • 飞书卡片更新必须低于单消息频控，并控制卡片体积。
  • 服务端要能审计 inbound/outbound/system/permission 消息。
  • 当前能力优先保证本机、局域网、Tailscale、隧道场景可用。

2. 技术方案调研对比

调研分为三个层面：消息入口、Claude Code 执行通道、进度与回复展示。核心取舍不是“哪一个方案最先进”，而是“哪个方案在当前用户环境里可发布、可恢复、可观测”。

方案 能解决什么 优点 限制 当前评分 结论
飞书自定义机器人 webhook 向群里发通知。 配置简单，适合 hook 事件通知。 只能作为输出入口，不能可靠接收用户消息和维护双向对话。 低 保留通知用途，不用于远程输入。
飞书自建应用长连接 接收私聊/群聊消息，发送文本和卡片。 官方支持消息事件，适合 IM bot；不需要公网 request URL。 需要应用权限、机器人能力、事件订阅和 app secret；事件处理要快速返回。 高 作为飞书输入主路径。
Claude Code MCP channel 把外部事件注入已经运行的 Claude Code session。 架构最贴近“继续当前会话”，支持双向 reply tool 和 permission relay。 官方仍标记为 research preview，需要版本、认证和组织开关；事件只在 session 打开时到达。 中 实现并保留，但不作为默认发布主链路。
Claude CLI print 模式 以命令行方式执行 prompt，输出 JSON/stream-json。 当前可用性最好；支持 --resume；stream-json 能聚合进度。 每条远程消息是一次 CLI 进程，不等价于把消息注入正在交互的 TTY。 高 作为 v2.0.0 默认主链路。
只依赖 hooks/dashboard 观察本机 Claude Code 执行过程。 已有基础扎实，适合做接收和展示。 hooks 不是输入通道，无法主动把远程消息送给 Claude。 中 作为可观测层，不承担消息投递。
官方 Claude Web/App/Slack 官方远程或团队聊天入口。 体验成熟，维护成本低。 不能直接满足飞书、本机代理、本地项目目录、私有 Dashboard 和中文办公 IM 场景。 中 不是本项目目标替代方案。
维度 CLI fallback MCP channel 项目取舍
本机可用性 只依赖 Claude CLI，可立即验证。 依赖 Claude Code channels 可用、认证和组织开关。 默认 CLI。
会话连续性 通过 --resume sessionId 继续。 直接注入正在运行的 session。 channel 更理想，但 CLI 足够支撑 v1 体验。
进度观测 stream-json、partial messages、hook events。 需要 channel 内事件和 Claude tool reply 组合。 CLI 进度更容易落地。
飞书体验 单卡片进度 + 最终文本。 同样可复用进度卡片聚合器。 展示层与执行器解耦。
失败恢复 进程退出即可标记 failed，错误清晰。 要处理 instance 心跳、SSE 断线、session 关闭。 v2 先确保失败可解释。

方案决策

采用“飞书自建应用长连接 + llm-proxy Remote Bridge + CLI fallback 主执行器 + MCP channel 可选增强”的组合方案。 这不是否定 channel，而是把 channel 放在合适的位置：当它在用户环境可用时提供更强的会话注入能力；在不可用时不影响远程对话的基本可用性。

3. 技术架构设计

架构分为入口层、控制层、执行层、展示层和存储层。入口层负责接收 Web/飞书消息，控制层负责 thread 选择、权限和路由，执行层负责 CLI 或 channel 投递，展示层负责 Dashboard SSE 和飞书卡片/文本，存储层负责审计和恢复。

入口层

Web Dashboard

POST /api/remote/send 新建或继续远程对话。

输入消息

飞书自建应用

WSClient 长连接接收 im.message.receive_v1。

llm-proxy 控制层

Remote Service

sendRemoteMessage 创建 thread/message，校验 cwd，选择执行器。

SQLite

保存 threads、messages、cards、instances、permissions。

Progress Aggregator

聚合 stream-json/hook/tool 事件，生成可展示快照。

执行与展示层

CLI fallback

claude -p --output-format stream-json --verbose。

MCP channel

notifications/claude/channel + remote_reply。

输出

Dashboard SSE、飞书进度卡片、最终普通文本。

运行时主链路：CLI fallback

1

接收消息

Web API 或飞书长连接收到 prompt，解析命令、sender、chat 和 source message。

2

选择 thread

新建时按 cwd 创建，继续时按 shortId、最近 active thread 或 sourceThreadId 查找。

3

写入消息

创建 inbound remote_message，初始化 progress snapshot 和飞书 progress card。

4

执行 Claude

根据 deliveryMode=cli 进入 dispatchToCli 并 spawn Claude CLI。

5

聚合进度

逐行解析 stream-json，更新 phase、events、tools、answerPreview 和 card。

6

发送回复

Stop hook 或 result 出现最终文本后，写 outbound 并向飞书/Web 推送。

数据模型

表 职责 关键字段 设计原因
remote_threads 远程会话主实体。 shortId、source、cwd、claudeSessionId、channelInstanceId、status 把飞书/Web thread 映射到本机 cwd 和 Claude session。
remote_messages 远程消息流水。 direction、sourceMessageId、sourceUserId、status、error 审计 inbound/outbound/system/permission，并支持失败追踪。
remote_message_cards 飞书进度卡片映射。 inboundMessageId、providerMessageId、lastSnapshot、lastPatchedAt 确保每条飞书入站消息只更新自己的卡片，不互相覆盖。
remote_channel_instances MCP channel 进程心跳。 id、pid、cwd、claudeSessionId、lastSeenAt channel 模式下选择在线 instance，识别 session 是否还活着。
remote_permissions 远程权限审批。 requestId、toolName、inputPreview、behavior、status 把 Claude Code permission_request 转成 Web/飞书可审批事件。

状态机

pendingthread 已创建
queued等待执行器
runningClaude 正在处理
waiting_permission等待审批
done最终回复已产出
failed执行或投递失败

message 状态独立于 thread 状态，使用 queued、sent、delivered、failed 表示投递生命周期。

4.1 飞书接入实现

输入侧
  • startFeishuRemoteBridge 使用飞书 SDK WSClient 建立长连接。
  • handleFeishuText 过滤群聊触发条件、校验 allowlist、解析文本。
  • parseFeishuRemoteCommand 支持 /help、/status、/projects、/new、/continue、/use。
  • parsePermissionReply 支持 同意 id、拒绝 id、yes id、no id。
输出侧
  • 入站消息创建 remote_message_cards，发送一张共享卡片。
  • buildFeishuProgressCard 固定 wide_screen_mode 和 update_multi。
  • patch 使用单卡片队列、最小间隔和 trailing update 合并，降低频控风险。
  • 最终回复通过 sendFinalReplyText 发送普通文本，超长内容分段。

4.2 CLI fallback 实现

CLI fallback 的核心是 runClaudePrint。它不是简单执行一条命令，而是把 Claude CLI 包装成可观测、可恢复、可失败归因的执行器。 

claude -p \
  --output-format stream-json \
  --verbose \
  --include-partial-messages \
  --include-hook-events \
  [permission args] \
  [--resume sessionId] \
  -- <prompt>
  • --verbose 是 stream-json 模式下必要参数，避免 Claude CLI 报错。
  • --include-partial-messages 让进度卡片能展示生成中摘要。
  • --include-hook-events 让 Stop hook、tool/hook 事件进入 reducer。
  • 当 Stop hook 的 last_assistant_message 或 result 事件出现时，提前发送最终回复，减少飞书等待。
  • 进程关闭后仍会解析最终 result，作为 early final 的兜底。

4.3 MCP channel 实现

channel 路径已经实现，但默认不启用。它的核心是一个本地 MCP server：claude-proxy-channel。

Claude Code session -> starts claude-proxy-channel MCP server -> channel registers instance to llm-proxy -> channel opens /api/remote/channel/events SSE -> llm-proxy pushes remote message event -> channel sends notifications/claude/channel -> Claude handles the message -> Claude calls remote_reply tool -> llm-proxy writes outbound message and pushes Web/Feishu reply
  • 能力声明包含 experimental["claude/channel"]。
  • 权限 relay 预留 experimental["claude/channel/permission"]。
  • 内部 SSE 不在 URL query 放 token，而是使用 X-Remote-Bridge-Token。
  • remote_reply 必须携带 remote_thread_id，服务端校验 instance/thread 绑定。
  • 安装命令 claude-proxy channel install --scope project --port 1998 写入项目 .mcp.json。

4.4 安全边界

执行目录

所有新建远程对话都要通过 allowedCwds 和 defaultCwd 校验，避免远程任意目录执行。

远程 API

channel/internal API 使用 remoteBridge.authToken。Dashboard 走同源边界，配置响应隐藏 token。

飞书权限

支持 allowedUserIds；群聊只处理 @bot 或回复 bot 链路，降低误触发和群成员滥用风险。

5. 风险与演进

风险 当前处理 后续演进
Claude Code channels 不可用或协议变动。 默认 CLI fallback，channel 仅在 auto/channel 模式使用。 跟随官方 channels 稳定后，逐步把 auto 设为推荐模式。
CLI fallback 不是真正注入现有 TTY。 用 --resume sessionId 保持上下文连续，并写回 thread 的 claudeSessionId。 channel 可用后，用在线 instance 接管长会话场景。
飞书卡片 patch 频控和大小限制。 单卡片队列、最小间隔、合并 trailing update、失败只降级提示一次。 引入快照 diff 和动态降采样，进一步减少 patch。
公网暴露 Dashboard 的安全风险。 文档要求使用带鉴权隧道/反代，API token 与 cwd allowlist 限制执行范围。 后续补账号体系、一次性访问链接和更细粒度 RBAC。
权限审批卡在远程链路。 channel permission relay 已预留，CLI 侧依赖 Claude CLI permission 参数。 统一 Web/飞书审批中心，支持超时、撤销和审批审计。
